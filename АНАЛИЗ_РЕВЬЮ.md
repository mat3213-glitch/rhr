# RHR — анализ и рекомендации

> Ревью кодовой базы от 2026-06-25. Разобрано 62 файла, ~6.5k строк Python, прогнан тест-сет.
> Код не менялся — это только отчёт. Приоритеты: 🔴 ломает работу сейчас · 🟠 надёжность · 🟡 гигиена.

## TL;DR

Архитектура крепкая: мультипликативный скоринг (один near-zero фактор топит кандидата — то что нужно),
чистый registry коллекторов, repo-as-database, идемпотентный пайплайн, 327 тестов.
Но: **reddit собирает ноль**, **в наборе 1 падающий тест**, **issues создаются без labels** (ломает канбан),
**воркфлоу пушат БД в `main` без rebase** (гонка → потеря снапшота), **нет CI на тесты**.

Что чинить первым:
1. `min_score` reddit + порог трекинга (2 строки) — чинят пустой сбор и красный тест.
2. `tests.yml` CI — чтобы регрессии ловились до мёржа.
3. `git pull --rebase` во все три пушащих воркфлоу — иначе данные теряются молча.

## 🔴 Баги, которые сейчас молча ломают сбор

### 1. Reddit-коллектор выкидывает ВСЕ посты
`config/sources.yaml` ставит `min_score: 10`, а redlib RSS (`new.rss`) не отдаёт поле `score`
→ `entry.get("score")` всегда `None` → `score = 0` → `if score < min_score: continue`
режет каждый item. Reddit «enabled», но собирает **ноль**.

- **Файл:** `collectors/reddit.py:73-80`
- **Фикс (любой):**
  - `min_score: 0` для redlib в `sources.yaml`, **или**
  - парсить рейтинг из заголовка/summary редлиба, **или**
  - применять `min_score` только когда score реально пришёл:
    ```python
    if entry.get("score") is not None and score < min_score:
        continue
    ```

### 2. Падающий тест в наборе
`ПРОГРЕСС.md` обещает «305 pass», по факту **326 pass / 1 fail**.
`MIN_SCORE_TO_TRACK = 0.001` с комментом «show all for now» — порог занижен для отладки
и не возвращён, а `test_excludes_low_score` ждёт отсев `score=0.01`.
Сейчас в issue-трекер попадёт буквально любой кандидат.

- **Файл:** `pipeline/track.py:24` ↔ `tests/test_track.py:108`
- **Фикс:** вернуть порог (напр. `MIN_SCORE_TO_TRACK = 0.05`) **или** обновить тест под новое поведение
  (решить осознанно: «трекаем всё» или «только осмысленные»).

### 3. Issues создаются без labels — ломается канбан
`track()` зовёт `_create_issue(title, body, labels=[])` всегда с пустым списком,
хотя вся архитектура в README завязана на label-воронку (`L2-scored`, `L3-demand-check`…)
и дашборд читает issues по лейблам. Кандидаты создаются, но канбан-фильтрация их не видит.

- **Файл:** `pipeline/track.py` (вызов `_create_issue`)
- **Фикс:** прокидывать `labels=["L2-scored"]` (и завести лейблы в репо через `gh label create`).

## 🟠 Надёжность инфраструктуры

### 4. Гонка `git push` между воркфлоу
`scan` (каждые 30 мин), `rhr-telegram` (2×/день), `grok-x-search` (1×/день) — все коммитят
`data/rhr.db` и пушат в `main`, но у каждого **свой** concurrency-group, т.е. они могут наложиться.
Никто не делает `git pull --rebase` перед push → при пересечении один push отлетает
non-fast-forward и снапшот теряется.

- **Файлы:** `.github/workflows/scan-forums.yml`, `rhr-telegram.yml`, `grok-x-search.yml`
- **Фикс (любой):**
  - `git pull --rebase --autostash origin main` перед каждым push, **или**
  - общий `concurrency.group: git-write` на все три (сериализует запись), **или**
  - хранить БД как artifact, а не в git.

### 5. Нет CI, который гоняет тесты
327 тестов — и ни один воркфлоу их не запускает на push/PR. Именно поэтому падающий тест (#2)
проехал незамеченным. Самый дешёвый и окупаемый фикс.

- **Фикс:** добавить `.github/workflows/tests.yml`:
  ```yaml
  name: tests
  on: [push, pull_request]
  jobs:
    pytest:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
        - uses: actions/setup-python@v5
          with: { python-version: "3.12", cache: pip, cache-dependency-path: requirements.txt }
        - run: pip install -r requirements.txt
        - run: python -m pytest -q
  ```

### 6. WAL + git = риск потерять последние записи
Схема включает `PRAGMA journal_mode = WAL`, а `.gitignore` исключает `*.db-wal`.
Пока процесс закрывает соединение перед коммитом — чекпойнт проходит и всё ок,
но это неявная гарантия.

- **Файл:** `data/schema.sql:5`
- **Фикс:** явный `conn.execute("PRAGMA wal_checkpoint(TRUNCATE)")` перед коммитом снапшота,
  либо `journal_mode=DELETE` для коммитимой БД.

### 7. `scan` каждые 30 мин коммитит БД → 48 коммитов/день
Repo-as-database раздуется за недели.

- **Фикс:** реже коммитить, squash-снапшоты, или artifact-хранилище.

## 🟡 Мелочи / гигиена

- **Doc↔impl рассинхрон по Telegram:** README говорит «Telegram нужен self-hosted runner»,
  а реальный коллектор (`collectors/telegram.py`) — анонимный скрап `t.me/s/` и крутится на
  `ubuntu-latest`. С шаренных GitHub-IP `t.me` может ловить rate-limit. Либо обнови доку,
  либо вынеси на свой раннер (под это есть Android/Termux-прокси).
- **Мёртвые RSS-плейсхолдеры** в `config/sources.yaml`:
  `substack.com/feed/note/c/o5a5f # placeholder` и `feedburner/_psfk` — проверь живость.
- **`ПРОГРЕСС.md` в корне** (кириллица в имени) — лучше слить в `PLAN.md`/`CHANGELOG.md`;
  README ссылается на `PLAN.md`, которого в репо нет.
- **Безопасность — чисто:** `.env` в `.gitignore` ✅, по `git grep` ни одного закоммиченного
  токена (`ghp_/xai-/sk-`) ✅. Приватность репо оправдана, утечек нет.
